## Accessibility & QA (Leandra, informed by internal IT conversation)
- Kent's IT team will handle **UAT** internally via their own UAT test-case process; Kent's brand cohort team will likely execute the actual testing against IT-provided cases. Leandra considers this covered without needing a dedicated accessibility specialist for this project.
- Kent currently has only a minimal (two-sentence) accessibility statement. Leandra had not seen a more robust example before this call (contrasted with Kent's own WellJoy site, which has a fuller accessibility menu).
- Kayla's guidance:
  - Offered to share resources on what a stronger accessibility statement looks like — ideally naming a **specific target conformance level** (i.e., this project's stated target is **WCAG AA**) and describing concretely how that's represented on the site.
  - **Recommended against third-party accessibility overlay/plugin widgets** — cited specialist feedback that these can *degrade* the experience for actual assistive-technology users (analogized to using a misconfigured loaner computer) even though they look like a shortcut to non-disabled stakeholders.
  - E3's design work will target accessibility best practices, but **code-level accessibility (how the site functionally behaves for screen readers/assistive tech) is Codal's responsibility**, not E3's — explicitly flagged as something Codal needs to own.
- Action item: Johnnie and Leandra to have an internal conversation about how much to prioritize accessibility; Kayla can send sample accessibility-statement language if useful.
